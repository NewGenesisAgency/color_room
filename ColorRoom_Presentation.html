Le besoin

Problématique et cahier des charges

Le logiciel de recherche de la ColorRoom est trop complexe pour l'initiation. Comment rendre la salle accessible à des apprenants, de façon ludique et pédagogique ?
Objectif
  • Une série de serious games pour découvrir la lumière et la couleur
  • Utilisable depuis un simple navigateur, sans installation côté client
  • Adaptable à une seule plaque lumineuse
Contraintes
  • Déploiement sur Raspberry Pi 5 + SSD, conteneurisation Docker imposée
  • Fonctionnement hors-ligne (réseau local)
  • Latence des plaques (~dizaines de ms) à compenser
  • Création de jeux no-code par l'enseignant

React / Next.js / TypeScript  vs  JS + Node-RED

Pourquoi cette stack ?

E2
Piste initiale : Node-RED
  • Modèle flow-based : parfait pour l'automatisation, mal adapté à une appli multi-pages
  • Pas de composants visuels pour une UI/UX pédagogique riche
  • Flux vite illisibles à grande échelle, rendu HTML/CSS limité
Choix retenu : Next.js + React + TS
  • React : composants réutilisables, DOM virtuel et réconciliation, état réactif (re-rendu ciblé)
  • TypeScript strict : typage statique vérifié à la transpilation (tsc) + ESLint
  • Next.js : UI + Route Handlers natifs (pas d'Express séparé), un seul runtime Node
  • Next.js 16.2 · TypeScript 5.5 strict · better-sqlite3 11.5 · Three.js 0.160 · Docker

Docker · Raspberry Pi 5

Réseau et déploiement

  • Image Docker multi-stage (deps → builder → runner), base node:alpine, cible arm64
  • Application exposée sur le port 8080, supervisée par Portainer
  • Le Pi fournit le Wi-Fi local ColorRoom_WiFI (tablettes, PC)
  • API de supervision des dalles atteinte depuis le Pi ; CS-160 en USB
  • Base SQLite dans un volume Docker (persistance entre redémarrages)
  • Secrets (compte admin) isolés dans un fichier .env (hors versionnement Git)
  • Exposition de l'API de supervision via pare-feu Windows + portproxy (netsh, port 18080)

Ma partie · données

Base de données SQLite

E2
  • SQLite via better-sqlite3 (API synchrone, requêtes préparées = anti-injection SQL)
  • 7 tables : crg_users, crg_sessions, crg_classes, crg_class_members, crg_scores, crg_games…
  • Intégrité référentielle : clés étrangères ON DELETE CASCADE
  • Migrations idempotentes (CREATE TABLE IF NOT EXISTS) au démarrage
  • PRAGMA journal_mode=WAL + busy_timeout : lectures concurrentes, anti SQLITE_BUSY
CREATE TABLE IF NOT EXISTS crg_users (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,
  user_type TEXT DEFAULT 'apprenant',
  password_hash TEXT, niveau TEXT )

PRAGMA journal_mode = WAL;

Ma partie · sécurité

Authentification et données personnelles

E2
  • 3 rôles : administrateur (créé via .env), enseignant, apprenant
  • Hachage PBKDF2-HMAC-SHA512 : 100 000 itérations, sel 16 octets, clé dérivée 64 octets ; stockage au format sel:hash
  • Jeton de session aléatoire en cookie HttpOnly + SameSite=lax (7 j) : atténue vol de session (XSS) et CSRF
  • Données 100 % locales : tout reste sur le Pi, rien sur Internet
  • Minimisation (pseudo seul), droit à l'effacement en cascade, cloisonnement par rôle

Ma partie · intelligence artificielle

Puissance 4 et son IA minimax

E2
  • Grille 7×6 projetée sur les 42 dalles, à 2 joueurs ou contre l'ordinateur
  • IA 100 % hors-ligne, directement dans le navigateur
  • Algorithme minimax avec élagage alpha-bêta (anticipe jusqu'à 7 demi-coups / plies)
  • Heuristique d'évaluation (bonus alignements, malus menaces) + move ordering central pour maximiser l'élagage
  • 3 niveaux (vert / jaune / rouge) = profondeur 2 / 4 / 7
  • Décision quasi instantanée (~70 ms), fluide même sur tablette
// profondeur selon le niveau
const cfg = DIFF[difficulty]; // 2 / 4 / 7

const v = winsAt(g,row,col,AI)
  ? WIN_SCORE
  : minimax(g, cfg.depth-1,
    -Infinity, Infinity, false);

Ma partie · jeux en réseau

Les jeux multijoueur

E2
  • État de partie persisté en base, récupéré par polling périodique de la route /state
  • Plus simple à déployer sur Pi que des WebSockets
  • Spectre Chromatique : jusqu'à 8 joueurs, reproduire une couleur (score = distance)
  • Morpion en réseau : 2 joueurs (X/O), dalles rouge/bleu, détection de victoire
  • Multijoueur générique : chaque joueur règle un canal vers une cible commune
  • Heartbeat de présence ; jetons de session = UUID aléatoires ; hôte = siège 1

Mesure colorimétrique et chromaticité
